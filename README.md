
# Тестовое задание: Fullstack-разработчик (Keystone.js + Next.js/React)

## 📋 Описание задания

В рамках этого тестового задания предлагаем тебе разработать небольшое web-приложение, используя связку **Keystone.js, Prisma и GraphQL** для бекенда и **Next.js/React** для фронтенда. 

Тематика приложения — **«Каталог книг и отзывов»** (если хочешь, можешь выбрать схожую тематику: фильмы, посты в блоге и т.д.). Твоя задача — реализовать базовый CRUD-функционал, спроектировать схемы данных, настроить права доступа и связать клиентскую часть с API.

В этом репозитории мы подготовили стартовый шаблон с базовой конфигурацией

---



## 🛠️ Стек технологий

- **Backend**: [Keystone.js v6](https://keystonejs.com/), [Prisma](https://www.prisma.io/), [GraphQL](https://graphql.org/)
- **Frontend**: [Next.js](https://nextjs.org/), [Apollo Client](https://www.apollographql.com/docs/react)
- **База данных**: SQLite (встроена по умолчанию, но можно переключить на PostgreSQL/MySQL)
- **Тестирование**: [Jest](https://jestjs.io/) [Vitest](https://vitest.dev/) или на твой выбор

---



## 🚀 Что нужно реализовать



### 1. Моделирование данных (Keystone.js)
Создай схемы данных для следующих сущностей:
- **User (Пользователь)**: может регистрироваться, авторизоваться и оставлять отзывы.
- **Book (Книга)**: сущность, которую оценивают пользователи. *(Опционально можно добавить связанные сущности: Author, Genre).*
- **Review (Отзыв)**: текстовый комментарий и числовая оценка (например, от 1 до 5 звезд).

### 2. Авторизация и контроль доступа
- Настрой регистрацию и логин с использованием встроенного модуля аутентификации Keystone.js.
- Настрой **Access Control**: 
  - Гости могут только просматривать список книг и отзывы.
  - Только авторизованные пользователи могут оставлять свои отзывы.



### 3. Расширение логики и кастомизация API
- Реализуй **tracking-поля** (`createdAt`, `createdBy`, `updatedAt`, `updatedBy`) для сущностей. Настрой их автоматическое заполнение с помощью [Hooks](https://keystonejs.com/docs/guides/hooks).
- Сделай уникальную связку на уровне базы данных: один пользователь может оставить **только один отзыв** на конкретную книгу
- Напиши **кастомный GraphQL-запрос** с помощью [extendGraphqlSchema](https://keystonejs.com/docs/config/config#extend-graphql-schema). Например, запрос для вычисления средней оценки книги на основе всех её отзывов.


### 4. Клиентская часть (Next.js)
Реализуй минималистичный пользовательский интерфейс:
- Страницы регистрации и входа.
- Главная страница каталога со списком книг.
- Детальная страница книги с её описанием и списком отзывов.
- Форма для добавления нового отзыва (должна отображаться только авторизованным пользователям).

> ⚠️ **Важно:** Мы не оцениваем визуальный дизайн! Нам не важна красота интерфейса. Главное — чтобы данные корректно запрашивались с бекенда, отображались на странице, а формы работали без ошибок. Используй любые готовые UI-библиотеки (Tailwind, MUI и т.д.) для ускорения разработки.

### 5. Тестирование
- Напиши юнит-тесты на Jest/Vitest для проверки ключевого функционала: работа авторизации, корректность выполнения API-запросов, базовая валидация данных при создании отзыва.

## 📦 Как запустить стартовый проект

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/A14X4Y/test-assignment-fullstack
   cd keystone-starter
   ```
2. Установите зависимости:
   ```bash
   npm install
   ```
3. Скопируйте перменные окружения
  ```bash
  cp .env.example .env
  ```

4. Создайте БД для подключения
  ```bash
  docker compsoe up -d 
  ```
5. Запустите сервер разработки next и keystone:

  Keystone:
   ```bash
   npm run keystone:dev
   ```

Next.js:
   ```bash
   npm run next:dev
   ```
   
## 🎯 Критерии оценки
На что мы будем обращать внимание:

1. **Качество кода и архитектура**: читаемость, логичное разделение компонентов и работа с API.
2. **Работа с Keystone.js**: понимание концепций системы — правильная конфигурация схем, хуков, Access Control и кастомных резолверов.
3. **Работа с базой данных (Prisma)**: корректная настройка связей и ограничений (unique constraints).
4. **Frontend**: корректная интеграция с GraphQL-клиентом (например, Apollo), обработка состояний загрузки/ошибок. **Напоминаем: дизайн не оценивается.**
5. **Тестирование**: наличие и осмысленность тестов основного бизнес-процесса.
